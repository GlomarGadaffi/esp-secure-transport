# GloSSH — project status

_Last updated: 2026-06-11_

## TL;DR

GloSSH set out to be a **pure MIT / public-domain** secure-transport stack for
ESP-IDF — "TinySSH all the way down," with no Apache-2.0 (mbedTLS) and no
copyleft/wolfSSL dependency. The **secure-channel core is built and verified on
hardware**; the TLS **handshake is not**. The project is **paused** in favor of
its pragmatic sibling, [**GloSSHlet**](https://github.com/GlomarGadaffi/GloSSHlet)
(`littlessh`), for the reasons below.

## What works (verified on T-ETH-ELITE / ESP32-S3, IDF v6.0.1)

All of the following pass the on-device self-test (`glo_mtls_selftest`,
26/26 checks, 0 failures):

| Layer | Spec | Notes |
|-------|------|-------|
| Crypto primitives | — | X25519, IETF ChaCha20-Poly1305, SHA-256, HKDF, Ed25519 — on TinySSH's tinynacl + a hand-written IETF ChaCha20 |
| **M1 — TLS 1.3 key schedule** | RFC 8446 §7.1 | early/handshake/master secrets, traffic keys/ivs, finished keys. **Anchored to RFC 8448** published constants. |
| **M2 — Record layer** | RFC 8446 §5.2/5.3 | TLSCiphertext framing, per-record nonce, AEAD seal/open. Byte-checked vs the RFC 8439 KAT. |
| SSH server | — | TinySSH port (`components/tinyssh`), boots and listens on :22. Note: tinyssh's shell/exec path is non-functional on ESP (`execve` stub). |

KAT vectors were generated by a Python oracle proven against RFC 8448.

## What is NOT done

| Milestone | State |
|-----------|-------|
| **M3 — Handshake messages** | not started. ClientHello/ServerHello/EncryptedExtensions/Certificate/CertificateVerify/Finished, raw-key Ed25519 (RFC 7250). |
| **M4 — State machine + `glo_mtls_connect`** | not started. `glo_mtls_connect/read/write/close` in `components/glo_mtls/include/glo_mtls.h` are **stubs**. |
| Interop | untested — needs a RawPublicKey TLS 1.3 peer. |

To resume: the crypto + key schedule + record layer in `components/glo_mtls/`
are the foundation; M3 is pure message (de)serialization feeding the existing
`glo_ks`/`glo_record` APIs, M4 wires them to a socket.

## Why paused — the pivot to GloSSHlet

The founding premise was "avoid Apache-2.0 / mbedTLS." But **mbedTLS is already
compiled into every ESP-IDF image** (esp_tls, HTTPS, Wi-Fi enterprise, the PSA
API). So for an ESP-IDF target, avoiding it buys no smaller binary and no
reduced license surface — it only matters if "zero Apache in the tree" is a
*hard* requirement.

It wasn't. So for the actual goal (a clean, capable SSH console), **GloSSHlet /
`littlessh`** — ~1000 lines on PSA Crypto, hardware-accelerated, OpenSSH-
interoperable, full shell/exec — is the better tool, and `glo_mtls` is
redundant with mbedTLS's own TLS 1.3 client.

**GloSSH still has a reason to exist** if a genuinely pure MIT/PD, zero-Apache
stack is ever required (a non-ESP-IDF target, or a hard licensing constraint),
or specifically for tiny raw-public-key (RFC 7250) mTLS, which mbedTLS supports
poorly. The work here is correct and resumable.
